# Claude Code创始人的

**来源**: InfoQ中文

## 金句

💎 工程师正从代码生产者，转变为智能体系统的设计者。
💎 产品设计的核心是：你无法强迫人们做他们原本不会做的事。
💎 为未来六个月后的模型能力而设计，而不是围绕当下短板打补丁。

## 水下信息

• 在Meta（Facebook）早期，工程师晋升和项目成功的关键之一是具备“通才”能力，即不仅要会写代码，还要能进行产品开发、设计、用户研究，甚至利用午餐时间在食堂对非技术人员进行“观察性用户研究”来快速验证产品功能，这种跨职能实践是当时企业文化所鼓励和重视的。
• 在大公司（如Meta）进行跨部门合作时，如果两个团队的核心价值观和绩效目标（如一个追求快速发布和用户增长，另一个追求系统稳定性和性能指标）截然不同且根深蒂固，仅靠普通经理协调极难成功。真正的解决方案可能需要最高管理层（如扎克伯格级别）介入，通过调整组织架构来强制促成合作，而非仅靠基层努力。
• 在AI工程团队（如Anthropic的Claude团队）招聘时，一个重要的筛选标准是寻找有“副业项目”或对工作以外事物充满热情的“通才型”人才，例如周末做开源项目、甚至是对制作康普茶有热情的人，这表明其具备持续的好奇心和全面发展潜力，是团队文化偏好的内幕。
• 产品设计的核心“水下”原则是识别和利用“潜在需求”：不要试图强迫用户做新事，而是观察用户如何“滥用”现有产品（如在Facebook群组中自发进行物品买卖），分析这些数据，发现用户的真实意图，然后顺势设计新功能（如Marketplace）来更好地满足它，这是Facebook多个成功产品（Marketplace, Dating）背后的真实驱动力。
• 当工程师遇到一个技术难题（如觉得Redux状态管理太复杂）时，培养一种“直觉”来判断这是否是一个普遍问题至关重要。可以通过观察内部论坛、用户支持帖子来验证。如果确认是普遍痛点，自己动手创造一个更简单的解决方案（如开发Undux），不仅能解决自身问题，还可能成为在组织内展现影响力和获得成长的重要机会。

## 案例提取

1. 美国 Meta公司 Boris Cherny及其团队通过开发并推广“群组聊天”功能，为Facebook用户提供服务。该功能旨在将Messenger聊天记录与Facebook群组同步，以探索公共空间社交产品的新形态。项目最初在网页端启动，并验证了在Facebook群组内进行聊天的想法是可行的。团队通过午餐时间在食堂对工作人员进行观察性用户研究，以获取产品反馈。

2. 美国 Meta公司 Boris Cherny通过开发并内部推广Undux状态管理框架，为Facebook内部遇到Redux使用困难的工程师提供服务。他针对Redux的复杂性，开发了一个更简单的React状态管理框架。为了推广，他通过脚本分析用户群体，并为相关团队安排了二三十场技术讲座。Undux一度成为Facebook内部最流行的状态管理框架之一，早期使用者如通知团队的Jeff Case曾积极使用并共同调试。

3. 美国 Meta公司 Boris Cherny通过撰写书籍和创办线下聚会，为TypeScript开发者社区提供服务。他因对TypeScript着迷，在业余时间撰写了系统介绍该语言的书籍，并创办了当时世界上最大的TypeScript聚会。书籍深入介绍了条件类型、字面量类型等先进理念。聚会吸引了包括Node.js创始人Ryan Dahl在内的众多开发者参与，促进了技术交流。
