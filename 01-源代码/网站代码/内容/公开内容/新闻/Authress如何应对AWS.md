# Authress如何应对AWS

**来源**: InfoQ中文

## 金句

💎 一旦系统足够复杂，总有东西会坏，你必须为此做好准备。
💎 检测和响应变得和预部署测试一样重要。

## 水下信息

• 在构建高可用架构时，应避免完全依赖云服务商（如AWS）提供的默认健康检查服务（如Route 53健康检查），因为它们可能与您实际使用的核心服务共享底层故障域，导致在重大故障中失效。Authress的经验表明，需要构建自定义的、针对自身核心组件（数据库、消息队列、核心逻辑）和端到端延迟的监控方案来做出可靠的故障转移决策。
• 为了实现真正的弹性，故障转移策略不能仅停留在区域级别。更优的架构是结合使用CDN（如CloudFront）和边缘计算（如Lambda@Edge），将计算推向边缘，并实现从计算节点到数据库的多层级、渐进式故障转移（例如，先故障转移到本地备用数据库，再扩展到相邻区域），这比简单的DNS全局切换更精细、延迟更低。
• 在基础设施即代码（IaC）和自动化实践中，刻意避免过度追求代码的“DRY”（不重复）原则，有时为了系统简单性和可靠性，可以接受一定程度的重复。将基础设施拆分为简单、独立、变化频率低的小单元，比一个高度抽象、复杂且频繁变更的“干”代码库更稳定，能有效降低由自动化本身引入的故障风险。
• 对于复杂系统，预部署测试已不足以保障可靠性。必须将“检测和响应”能力提升到与测试同等重要的战略高度。系统总会以意想不到的方式故障，因此构建快速、准确的故障检测和自动化响应机制，是保持SLA的关键，这来自资深SRE（站点可靠性工程师）的实践经验总结。
• 处理云服务大规模中断时，一个有效的策略是主动最小化对云提供商“控制平面”服务（管理、配置、监控服务）的依赖。这些服务在区域级中断中同样可能不可用，将核心业务逻辑和数据平面的设计与控制平面解耦，可以增强在极端情况下的生存能力。

## 案例提取

1. 全球 AWS 基础设施 Authress 通过多区域部署与动态 DNS 故障转移策略为其身份和认证服务的用户提供服务。在2025年10月20日AWS发生十年来最严重的大规模中断期间，该策略确保了服务的高可用性。Authress 成功保持了其 SLA 可靠性承诺。这被 Hacker News 读者 rdoherty 评价为对其过去10年 SRE 职业生涯的最佳总结，强调了检测和响应的重要性。

2. 全球 AWS 基础设施 Authress 通过自定义健康检查与监控方案为其自身服务可靠性提供保障。该方案跨数据库、SQS 和核心授权器逻辑执行多项检查，并分析端到端请求延迟，以可靠确定六个主区域中是否出现问题并更新 DNS。此方案避免了依赖 AWS Route 53 默认健康检查可能带来的误判风险。首席技术官 Warren Parad 指出，这使他们能够可靠地确定主区域是否遇到问题。

3. 全球 AWS 基础设施 Authress 通过采用边缘优化架构（AWS CloudFront 和 Lambda@Edge）为其用户提供服务。该架构使服务更接近用户以减少延迟，并支持更健壮的、可隔离单个故障组件（如数据库）的故障转移策略。当某个区域数据库出现健康事件时，系统会自动故障转移到相邻区域。该设计解决了传统 DNS 故障转移策略难以隔离和替换单个故障组件的局限性。

4. 全球 AWS 基础设施 Authress 通过保持基础设施简单、小规模且重复（而非 DRY）的设计原则来管理其基础设施即代码（IaC）的风险。他们将基础设施拆分为单独的简单服务，减少变化频率以降低引入问题的机会。这种做法旨在避免因过度追求代码复用而带来的复杂性。首席技术官 Warren Parad 认为，变化不那么频繁会减少出现问题的机会。
