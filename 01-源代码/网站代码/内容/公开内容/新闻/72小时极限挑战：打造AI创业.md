# 72小时极限挑战：打造AI创业

**来源**: Google Cloud Tech

## 金句

💎 三支团队，三种截然不同的路径，始于同一个强大的框架。
💎 在72小时内，用代码直面每个初创公司都会遇到的巨大难题。
💎 不是改变模型，而是重新思考如何让智能流动起来。
💎 真正的挑战，是用现实场景去拷问你的智能体。

## 水下信息

• 使用 Google 的 Agent Development Kit (ADK) 进行云端部署时，选择 Cloud Run 并传递项目 ID 是一个快速部署的“捷径”，但实际部署过程可能异常缓慢（文中描述为“等了5年”），这揭示了在真实场景中，即使使用官方工具，部署也可能遇到意想不到的延迟和压力。
• 在构建多智能体系统时，存在不同的架构策略：一种是通过“超级简洁”的方式保持智能体间的上下文流动；另一种是构建“套娃式”的智能体，即主智能体调用子智能体，子智能体再调用顺序智能体。这些是实践中探索出的具体技术路径，而非公开文档中的标准方案。
• 实现多模态功能（如图像输入并转录文本）时，如果已经使用了 Gemini Flash 模型，可能无需更换模型，只需在代码中正确添加图像输入处理逻辑即可。这暗示了特定模型对多模态任务的开箱即用支持程度，是一个节省开发时间的关键点。
• 暴露工具 via Model Context Protocol (MCP) 和启用 Agent-to-Agent (A2A) 直接协作协议，在 ADK 框架下可能被有经验的开发者视为相对“简单”的挑战，只需添加包装器和少量导入语句即可。这指出了框架对某些高级协议的内置支持程度，降低了实现门槛。
• 模拟短时间内承受 1000 个并发用户访问端点的压力测试，被开发者描述为极具压力且可能导致部分功能失效（“第三个就是不工作”），这揭示了在有限时间内构建的 AI 智能体在实际负载下的脆弱性，是内部测试中才会暴露的真实性能瓶颈。
