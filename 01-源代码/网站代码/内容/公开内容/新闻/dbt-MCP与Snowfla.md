# dbt MCP与Snowfla

**来源**: InfoQ中文

## 金句

💎 Agent 的关键不在模型能力，而在是否存有一个可信的中间层。
💎 Agent 不是超级用户，而是受控但高效的协作者。
💎 真正的智能，建立在人类工程师已验证过的建模体系之上。

## 水下信息

• 将大语言模型（LLM）直接接入数据工程流程存在巨大风险，因为它可能“自由发挥”去“猜测”数据或SQL，导致不可控和不安全。真正的生产级解决方案需要一个可信的中间层（如dbt MCP Server）来翻译工程意图并约束模型行为。
• 在数据领域，有价值的智能体（Agent）必须是“行动型”的，而非仅仅是“问答型”。但让Agent行动的关键障碍不是模型能力，而是如何让它安全、可控地理解和操作分散在dbt项目、仓库元数据和团队约定中的复杂工程上下文。
• dbt MCP Server的核心角色不是简单地暴露API，而是充当一个“上下文协调器”。它结构化地向Agent呈现整个dbt项目的语义（模型、依赖、业务含义、操作权限），使Agent的“智能”建立在人类工程师已验证过的建模体系之上，而非绕过它。
• 一种避免智能体（Agent）成为“超级用户”从而失控的架构设计：让Agent不直接控制数据平台（如Snowflake），所有实际的数据查询和转换仍在平台原生执行体系内完成；Agent仅通过中间层发起符合规范的请求。这确保了治理、审计和性能成本管理不被绕过。
• 构建真正可投入生产的智能工作流（Agentic Workflow）的目标，不是“自动化一切”，而是在不破坏现有工程纪律的前提下，引入新的效率杠杆。这意味着最终的执行路径、校验方式和责任边界，仍需清晰地掌握在人类工程师手中。

## 案例提取

1. 全球 Snowflake 通过市场调查和数据分析为全球企业及IT专业人士提供投资回报洞察。调查了全球1900位企业IT专业人士，发现生成式AI投资的平均投资回报率高达41%。报告可供下载，提供了完整的投资回报分析。

2. 线上（Build 2025大会） dbt Labs 和 Snowflake 的实践者通过技术分享为技术从业者展示如何构建智能体工作流。核心案例是使用 dbt MCP Server 作为中介，让智能体理解、调用并执行 dbt 项目中的真实数据资产与逻辑，并最终运行在 Snowflake 上。分享强调工程现实与可操作性，而非概念先进性。该实践展示了一条让智能体基于成熟数据工程体系运作的可行路径。

3. 线上（Build 2025大会） dbt Labs的技术产品营销经理 Sarah Gawlinski 和高级经理 Jason Ganz 通过具体演示为观众讲解如何将大模型安全、可控地嵌入现有数据工作流。他们利用 dbt MCP Server 作为上下文协调器，为智能体提供结构化的 dbt 项目语义，使其能在预设的工程边界内安全操作。该方法避免了让大模型在“啃数据”和“啃SQL”层面自由发挥，而是将其智能建立在人类工程师已验证的建模体系之上。

4. 线上（Snowflake Discover活动） Snowflake 通过免费的线上实时互动技术活动为数据与AI从业者提供能力评估与知识分享。活动旨在帮助参与者全面评估数据与AI能力，深入了解如何更高效地管理、整合与分析数据。在4天时间内安排了18场技术干货分享，由来自亚太地区的一线技术专家亲自分享与讲解。活动提供了报名参与渠道。
