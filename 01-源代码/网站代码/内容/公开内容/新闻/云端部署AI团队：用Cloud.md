# 云端部署AI团队：用Cloud

**来源**: Google Cloud Tech

## 金句

💎 一个只存在于本地的功能不是功能，而是科学实验。
💎 我们独立构建它们，就能独立扩展它们。
💎 就像超市，结账队伍太长就多开收银台，不必为增加收银员而扩建停车场。
💎 你不再需要构建孤独的巨石应用。

## 水下信息

• 在本地开发环境中完美运行的AI代理架构，一旦部署到生产环境（如Cloud Run），其核心挑战并非功能实现，而是如何将本地“科学实验”转化为可公开访问、可伸缩的“功能”。这揭示了从开发到上线过程中，环境与可访问性是关键障碍。
• 采用微服务架构部署AI代理（如研究员、评审员、内容构建者）的核心优势在于可以独立伸缩。文中用“超市收银台”比喻：只需增加收银通道（对应某个繁忙的代理），而无需扩建停车场（对应整体扩容），这提供了成本效益极高的伸缩策略。
• 使用Google Cloud Run时，一个关键的成本优化特性是“可缩容至零”：当服务无请求时（例如夜间），计费停止，避免了为闲置的AI代理资源付费。这是将AI应用投入生产时控制运营成本的实际内幕技巧。
• 在演示或快速原型阶段，为了方便代理间通信，可能会临时使用“允许未经身份验证”的访问设置。但文章明确指出，在真实生产应用中，必须使用更严格的身份验证机制（如Cloud IAM），以确保只有协调器能调用特定代理（如评审员），这揭示了从演示到生产必须跨越的安全配置鸿沟。
• 升级单个AI代理（例如将内容构建者升级到新版的Gemini模型）时，只需更新并重新部署该代理，而不会导致整个系统停机。这体现了微服务架构在维护和迭代AI能力时的实际优势：实现零停机更新，降低了生产环境迭代的风险和复杂度。
